Використання sleep(5) було обрано як тимчасове рішення для стабілізації тестів на динамічних елементах Angular, оскільки деякі елементи (наприклад, mat-mdc-select-arrow) змінюють свої стани або класи після завантаження DOM, що спричиняло StaleElementReferenceException або TimeoutException при використанні стандартних очікувань.

В деяких випадках не працювала коректно, приклад: 
```python 
driver.maximize_window()
    driver.get(BASE_URL)
    print(driver.title)
    sleep(5)
    
    filter_viewbox_xpath = "//div[@class='mat-mdc-select-arrow ng-tns-c3393473648-0']"
    filter_viewbox = driver.find_element(By.XPATH, filter_viewbox_xpath)
    
    assert filter_viewbox.is_displayed(), "Filter viewbox is not displayed"
    
    filter_viewbox.click()
```

**Методологія вибору селекторів** 
- XPath з функцією contains(): Використано для знаходження елементів, де частина класу є динамічною (наприклад, Angular додає префікси ng-).

        Приклад з ТС-03: //div[contains(@class, 'container-img')] — це дозволяє знайти іконку пошуку, навіть якщо її повний клас зміниться.

- Пошук за ID: Використано для елементів випадаючих списків (Material Design), де ID є найбільш прямим способом звернення.

        - Приклад з ТС-02: By.ID, 'mat-option-4'.

- Пошук за атрибутами: Використано `placeholder='Search'`, оскільки це найбільш людиночитаний та стабільний атрибут для полів вводу.

# ТСА-01 "Реєстрація нового користувача" :smile:
### Тестування форми створення акаунта з перевіркою валідації та обробкою системних помилок.
Додано перевірку на виявлення помилки. Так як реєстрація пройшла успішно, сайт не видає помилку.


 ```python   
    try:
        
        error_message_xpath = "//div[@class='alert-general-error']"

        short_wait = WebDriverWait(driver, 5)

        error_element = short_wait.until(lambda d: d.find_element(By.XPATH, error_message_xpath))

        print("Знайдено помилку")

    except:

        print("Помилки не виявлено.") 
```

# ТС-02 "Фільтрація події за тегами" :blush:
### Перевірка динамічного списку фільтрів та коректності відображення елементів після вибору тега

У цьому тесті було реалізовано роботу з випадаючими списками Angular Material:

1. Взаємодія: Спочатку виконується клік по filter_viewbox (стрілка фільтра), щоб розгорнути список.

    - Верифікація: Використано assert category_filter.is_displayed(), щоб переконатися, що контейнер зі списком (role='listbox') дійсно з'явився на екрані перед вибором опції.

Особливість: Робота з mat-option вимагає коротких пауз (sleep(2)), оскільки анімація появи списку займає певний час.

# ТС-03 "Пошук події за назвою" :laughing:
### Емуляція пошукового запиту через глобальний пошук та верифікація знайденого контенту.

Тест демонструє ланцюгову взаємодію з прихованими елементами:

- Логіка: Спочатку натискається іконка лупи, що викликає появу інпуту.

- Емуляція дій користувача: Замість використання окремої кнопки "Знайти", використано символ переносу рядка \n у методі send_keys, що імітує натискання клавіші Enter.

- Обробка результатів: Використано блок try-except разом з assert. Це дозволяє тесту перевірити наявність картки image-container, але не "падати" з критичною помилкою, якщо за конкретним запитом івентів не знайдено (гнучкий тест).

Реалізовано блок перевірки на наявність івентів try-except:

```python 
try:
        result_card = driver.find_element(By.XPATH, "//div[(@class='image-container')]")
        assert result_card.is_displayed(), "Result card is not displayed"
        if result_card.is_displayed():
            print("Є івенти!")
    except:
        print("Немає івентів!")
```

## 📝Обробка помилок та стабільність

Для забезпечення стабільності тестів було застосовано комбінований підхід: 
- Явні очікування: створення об'єкта `WebDriverWait` для підготовки до роботи з елементами.
- Статичні паузи: Використання `sleep()`, що виявилось необхідним для коректної роботи з анімаціями Angular Material, які не завжди перехоплюються стандартними очікування Selenium.
- Твердження: Кожен ключовий етап тесту супровождається перевіркою `is_displayed()`, що гарантує наявність елемента в полі зору користувача перед взаємодією.
